Hands-on Elixir & OTP: Cryptocurrency Trading Bot

Chapter 22 Back to the Monolith

In the last chapter, we proved that the pure core of our strategy is cheap to test - no mocks, no state, just inputs and outputs. Our code is in good shape. But the project that wraps it? Not so much.

We’ve been carrying the umbrella structure since Chapter 1, and it was a reasonable bet at the time - isolate apps now, extract to services later if needed. Three years and twenty-two chapters later, that “later” never arrived. What did arrive was configuration drift across configuration files, three separate databases for what is logically one system, and a deployment story that multiplies every operational concern by the number of child apps.

The industry has largely come around to this view. Even teams that championed microservices early have written about the return to monoliths once the coordination cost outweighed the isolation benefits (see the “Majestic Monolith” pattern popularised by DHH at Basecamp). The lesson isn’t that umbrellas or microservices are wrong - it’s that they have a cost, and you should only pay it when the problem demands it.

In this chapter, we’ll consolidate everything into a single Phoenix application. It’s a lot of file-moving and renaming - mechanically simple, but strategically important. By the end, we’ll have one repo, one database, one config surface, and a clean foundation for deployment and clustering in the chapters ahead.

22.2 Reverting the course to the monolith

Our plan is straightforward: create a fresh Phoenix application side-by-side, then reintegrate each umbrella app one at a time - starting with core(the shared dependency), then binance_mock, streamer, naive, data_warehouse, and finally indicator.

22.3 Creating a new Phoenix app

First, let’s make sure that we have the latest version of the Phoenix application generator installed:

$ mix archive.install hex phx_new

Note: At the time of writing this chapter, the current version of the Phoenix framework is 1.8.3.

With this taken care of, we can progress with the creation of a new Phoenix application and a database:

$ mix phx.new hedgehog
...
$ cd hedgehog
$ mix ecto.create

With the Phoenix app skeleton in place, we can move on to reintegrating the umbrella apps’ contents into it.

22.4 Reintegrating the core app

We will start with the core app, as it is used/referenced by other apps in the umbrella.

It contains just a single module called Core.Struct.TradeEvent (inside the apps/core/lib/core/struct/trade_event.ex file) which we will rename to Hedgehog.Exchange.TradeEvent inside the newly created Phoenix app (we need to create a new exchange directory inside the /lib/hedgehog and paste the updated trade_event.ex file inside).

Besides the TradeEvent module, the core application supervises the PubSub supervision tree. After a quick check inside the /lib/hedgehog/application.ex file, we can confirm that Phoenix already uses PubSub:

  # /lib/hedgehog/application.ex
    children = [
      ...
      {Phoenix.PubSub, name: Hedgehog.PubSub},
    ...

Furthermore, the phoenix_pubsub dependency is already included in the mix.lock file.

The above change finishes merging the core app into our new Phoenix app. We should be able to run our app:

22.11 Final thoughts

This was the longest and most mechanical chapter in the book - and that’s fine. Here’s what we accomplished:

  • Consolidated six umbrella apps into a single Phoenix application
  • Eliminated three separate databases in favour of one shared Hedgehog.Repo
  • Unified configuration into a single config.exs surface
  • Preserved all existing tests - both unit and integration - without modification to their logic
  • Renamed modules to fit a cleaner namespace(Strategy.Naive.Formula, Data.Collector, Data.Aggregator)

None of the above changed what our system does. It still streams trade events, executes the naive strategy, stores data, and aggregates OHLC candles. What changed is how much effort it takes to operate: one release, one database, one configuration tree.

This is worth calling out explicitly: the best refactors often look boring from the outside. No new features, no clever abstractions - just less surface area for things to go wrong.

So, What’s Next?

We have a single Phoenix application, but we’re still tied to Postgres running in Docker - a dependency that adds operational weight and makes each node dependent on the network for every write. In the next chapter, we’ll migrate from Postgres to SQLite, giving each trading node its own local database. This sets us up for the real prize: spreading our trading across multiple independent Elixir nodes in an Erlang cluster, where each node is fully self-contained yet still connected.
