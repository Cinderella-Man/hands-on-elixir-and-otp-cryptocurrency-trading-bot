Hands-on Elixir & OTP: Cryptocurrency Trading Bot

Chapter 24 Clustering on the Open Internet

In the last chapter, we replaced Postgres with SQLite - each trading node now carries its own database as a single file. We set up Litestream for continuous backups and removed every external infrastructure dependency. The application is fully self-contained.

But self-contained nodes sitting in isolation aren’t very useful. We want to run multiple trading nodes - potentially on different machines, in different data centres, maybe even different continents - and still have a unified view of what’s happening across the fleet. Which symbols are being traded? How many open positions exist? What’s the total order volume? Right now, answering any of these questions means SSH-ing into each machine and running queries by hand.

Erlang was built for exactly this problem. The BEAM has native support for connecting nodes into a cluster, where any process on any node can communicate with any process on any other node as if they were local. This is not an add-on or a library feature - it’s baked into the runtime. Two Erlang nodes that know each other’s names and share a secret cookie can form a cluster with a single function call.

In this chapter, we’ll build an OTP release, deploy it to plain virtual machines(no containers), configure Erlang distribution for secure communication over the open internet, wire up libcluster for automatic cluster formation, and build a module that queries data across all connected nodes. By the end, we’ll have a fleet of independent trading bots that can see each other.

24.1 Objectives

  • building an OTP release for deployment
  • deploying to virtual machines without containers
  • understanding Erlang distribution: node names, cookies, and EPMD
  • securing inter-node communication on the open internet
  • configuring libcluster for automatic cluster formation
  • querying data across a dynamic set of nodes

24.2 Building an OTP release

An OTP release is a self-contained package of your application, its dependencies, the Elixir runtime, and the Erlang VM - everything needed to run without installing Elixir or Erlang on the target machine. Elixir’s built-in mix release handles this for us.

24.2.1 Configuring the release in mix.exs

Before generating any files, let’s configure the release itself. Every release is tied to a version - this is the version in your mix.exs, and it matters more than you might think. OTP uses this version to track what’s running on each node, and later(when we add hot code upgrades) to compute the differences between the old and new code.

Make sure your mix.exs has an explicit version:

# /mix.exs
defmodule Hedgehog.MixProject do
  use Mix.Project

  def project do
    [
      app: :hedgehog,
      version: "0.1.0", # <= this version drives the release
      ...
      releases: releases()
    ]
  end

  ...

  defp releases do
    [
      hedgehog: [
        include_executables_for: [:unix],
        strip_beams: false, # <= added - keep debug info for upgrades
        steps: [:assemble, :tar] # <= added - produce a tarball
      ]
    ]
  end
end

Two things to note about this configuration. strip_beams: false preserves debug information and module metadata in the compiled .beam files. Without this, OTP cannot generate the instructions it needs when upgrading code on a running node - we’ll use this in the next chapter when we introduce hot code reloading. The cost is slightly larger beam files(typically 20-40% bigger), but for a trading bot this is negligible.

steps: [:assemble, :tar] tells mix release to produce a compressed tarball alongside the normal release directory. This tarball is what we’ll ship to production servers. Deploying a tarball rather than syncing a directory is important: it gives us a clean, versioned artifact that we can unpack on the server without destroying the previous version’s files. That matters for rollbacks, and it matters even more when we add hot code upgrades.

24.2.2 Generating release configuration files

Generate the template files that control how the release behaves:

mix release.init
* creating rel/vm.args.eex
* creating rel/remote.vm.args.eex
* creating rel/env.sh.eex
* creating rel/env.bat.eex

This creates four files in the rel/ directory. We care about two of them: env.sh.eex(sets environment variables before the VM starts) and vm.args.eex(passes flags directly to the Erlang VM).

24.2.3 Configuring the node name and cookie

For nodes to find and trust each other, every node needs two things: a unique name and a shared secret called a cookie. We’ll set both through environment variables in rel/env.sh.eex:

# /rel/env.sh.eex
export RELEASE_DISTRIBUTION=name
export RELEASE_NODE=${NODE_NAME:-hedgehog@127.0.0.1}
export RELEASE_COOKIE=${RELEASE_COOKIE:-$(cat /home/hedgehog/.erlang.cookie)}

A few things to note here. RELEASE_DISTRIBUTION=name tells the VM to use fully qualified names(also called “long names”) - this is required when nodes run on different machines. The alternative, sname(short names), only works within a single hostname or local network and cannot cross network boundaries.

RELEASE_NODE sets the node’s identity. The format is name@host, where host must be an IP address or a DNS name that other nodes can resolve. We default to hedgehog@127.0.0.1 for local development, but in production you’ll set NODE_NAME to something like hedgehog@203.0.113.10.

RELEASE_COOKIE is the shared secret. We read it from a file by default, but you can also set it directly as an environment variable. The cookie must be identical on every node that should form a cluster.

24.2.4 Constraining the distribution port range

Erlang uses a daemon called EPMD(Erlang Port Mapper Daemon) to map node names to ports. EPMD itself listens on port 4369, and each node picks a random high port for its actual inter-node traffic. This is a problem for firewalls - you can’t whitelist a random port.

We fix this by telling the VM to use a specific port range. Add the following to rel/vm.args.eex:

# /rel/vm.args.eex
-kernel inet_dist_listen_min 9100
-kernel inet_dist_listen_max 9155

This constrains the distribution port to a known range. For a single node per machine, a range of one port(e.g. 9100-9100) is sufficient. A wider range accommodates multiple nodes on the same machine during development or testing.

24.2.5 Running database migrations in a release

In development, we run mix ecto.migrate. But a release doesn’t have Mix - it’s a compiled artifact. Phoenix projects include a release module that provides migration support. If your project doesn’t have one, create it:

# /lib/hedgehog/release.ex
defmodule Hedgehog.Release do
  @app :hedgehog

  def migrate do
    load_app()

    for repo <- repos() do
      {:ok, _, _} =
        Ecto.Migrator.with_repo(repo, &Ecto.Migrator.run(&1, :up, all: true))
    end
  end

  def rollback(repo, version) do
    load_app()
    {:ok, _, _} = Ecto.Migrator.with_repo(repo, &Ecto.Migrator.run(&1, :down, to: version))
  end

  defp repos do
    Application.fetch_env!(@app, :ecto_repos)
  end

  defp load_app do
    Application.load(@app)
  end
end

Notice that load_app/0 uses Application.load/1 - not Application.ensure_all_started/1. The difference is critical. load/1 reads the application’s configuration without starting any processes. ensure_all_started/1 would boot the entire supervision tree - the Repo, Litestream, Phoenix endpoint, trading processes - all before the migration has run. That means Litestream would try to connect to S3, the Repo would try to open a database that might not have the right schema yet, and you’d need every production environment variable set just to run a migration. Ecto.Migrator.with_repo/2 handles starting just the Repo for us, which is all a migration needs.

After deploying, run migrations with:

bin/hedgehog eval "Hedgehog.Release.migrate()"

24.2.6 Building the release

Build the release for production:

MIX_ENV=prod mix release
...
* assembling hedgehog-0.1.0 on MIX_ENV=prod
* using config/runtime.exs to configure the release at runtime
* building hedgehog-0.1.0.tar.gz
...
Release created at _build/prod/rel/hedgehog

Because we configured steps: [:assemble, :tar], the build produces both the release directory and a tarball at _build/prod/rel/hedgehog/hedgehog-0.1.0.tar.gz. The tarball is what we’ll ship to production. It contains everything - the ERTS runtime, your compiled code, all dependencies - in a single file named after your application version.

Note on cross-compilation: The release must be built on the same OS and architecture as the target machine. If you develop on macOS but deploy to Linux, you need to build the release on a Linux machine(or in a Linux VM/CI pipeline). You cannot build on x86_64-apple-darwin and deploy to x86_64-unknown-linux-gnu - the ERTS binary won’t run.

24.3 Deploying to a virtual machine

We’re deploying to plain VMs - no Docker, no Kubernetes, no orchestrator. Just Linux machines with an SSH connection. This is the simplest production setup and it maps directly to our architecture: one machine, one node, one database.

24.3.1 What you need on the target machine

The target machine needs surprisingly little:

  • A Linux distribution(Ubuntu, Debian, etc.)
  • A user account to run the application(don’t run as root)
  • The libncurses and libssl libraries(required by ERTS)
  • Directories for the application and database

Set up the target machine:

# On the target machine
sudo apt-get update && sudo apt-get install -y libncurses5 libssl-dev
sudo useradd -m -s /bin/bash hedgehog
sudo mkdir -p /opt/hedgehog /data/hedgehog
sudo chown hedgehog:hedgehog /opt/hedgehog /data/hedgehog

24.3.2 Deploying the release

Copy the tarball to the target machine and unpack it:

# From the build machine
scp _build/prod/rel/hedgehog/hedgehog-0.1.0.tar.gz \
  hedgehog@your-server:/opt/hedgehog/
# On the target machine, as the hedgehog user
cd /opt/hedgehog
tar -xzf hedgehog-0.1.0.tar.gz
rm hedgehog-0.1.0.tar.gz

After unpacking, /opt/hedgehog contains the standard OTP release layout:

/opt/hedgehog/
├── bin/
│   └── hedgehog          # CLI entry point
├── erts-<version>/       # Erlang runtime
├── lib/
│   ├── hedgehog-0.1.0/   # your compiled code (versioned!)
│   ├── ecto-3.x.x/       # dependencies (versioned)
│   └── ...
└── releases/
    ├── 0.1.0/            # release metadata for this version
    │   ├── sys.config
    │   └── vm.args
    ├── RELEASES          # tracks installed versions
    └── start_erl.data

Notice how everything is versioned - lib/hedgehog-0.1.0/, releases/0.1.0/. This is the OTP convention and it matters. When we deploy version 0.2.0 later, the new directories sit alongside the old ones rather than replacing them. The running node can serve the old code while the new code is being installed - which is exactly what makes hot code upgrades possible.

Why not rsync? You might be tempted to use rsync -avz to push the release directory directly. This works for a first deploy but creates problems later: rsync overwrites existing files, which means the old version’s beam files are destroyed. If you ever need to rollback, or if you want to upgrade code on a running node without restarting it, you need both the old and new versions on disk simultaneously. Deploying via tarball preserves this property naturally - each version is a clean, self-contained snapshot.

24.4 Understanding Erlang distribution

Before we configure automatic clustering, it’s worth understanding what happens under the hood when two Erlang nodes connect. This knowledge will save you hours of debugging when things don’t work.

24.4.1 Node names

Every distributed Erlang node has a name in the format name@host. The name part is arbitrary - we use hedgehog. The host part must be something the other nodes can resolve: an IP address(hedgehog@203.0.113.10) or a DNS name(hedgehog@node1.example.com).

There are two naming modes:

  • Short names(-sname): The host part is just a hostname without a domain. Nodes using short names can only connect to other nodes on the same machine or the same local network. Fine for development, not for production across the internet.

  • Long names(-name): The host part is a fully qualified domain name or IP address. Required for nodes that need to connect across networks. This is what we configured with RELEASE_DISTRIBUTION=name.

A node using short names cannot connect to a node using long names - they must all agree on the naming mode.

24.4.2 The cookie

The cookie is a shared secret that acts as a first line of authentication. When node A tries to connect to node B, both nodes compare cookies. If they don’t match, the connection is refused. That’s it - there’s no encryption, no challenge-response, no certificate exchange. Just a string comparison.

This has an important security implication: anyone who knows your cookie and can reach your node’s distribution port has full access to your VM. They can execute arbitrary code, read your data, and shut down your application. We’ll address this in the security section.

Generate a strong cookie:

elixir -e "IO.puts(Base.encode32(:crypto.strong_rand_bytes(40)))"
W3VFHXQW2Q3EZBSGK7GWCPDCW36LVWBXD2NOYF54DXQLQYVW7YVA====

Use this value as RELEASE_COOKIE on every node. Don’t use short, guessable strings like hedgehog or secret - treat the cookie like a password.

24.4.3 EPMD

EPMD(Erlang Port Mapper Daemon) is a small service that runs on every machine hosting Erlang nodes. It listens on TCP port 4369 and maintains a registry of node names to distribution ports. When node A wants to connect to hedgehog@203.0.113.10, it first contacts EPMD on 203.0.113.10:4369 and asks “what port is hedgehog listening on?”. EPMD responds with the port number, and node A connects directly.

EPMD starts automatically when you start a distributed Erlang node. You don’t need to manage it separately, but you need to know it exists because:

  • Port 4369 must be reachable between nodes(or at least between the connecting node and the target’s EPMD)
  • The distribution port(which we constrained to 9100-9155) must also be reachable
  • EPMD itself has no authentication - anyone can query it to discover which nodes are running on a machine

Note on EPMD alternatives: Since OTP 25.1, Erlang supports running without EPMD entirely by using alternative service discovery mechanisms. For our setup with libcluster handling discovery, EPMD is still the simplest path. If you want to eliminate EPMD, look into the -epmd_module VM flag and custom EPMD modules - but that’s beyond the scope of this chapter.

24.5 Securing the cluster

Here’s the uncomfortable truth about Erlang distribution: it was designed for trusted networks. The cookie provides only a minimal gate - once past it, a connected node can execute arbitrary Erlang code on any other node in the cluster. There is no granular permission model. A connected node has the same power as a local IEx session.

This means running an Erlang cluster on the open internet without additional protection is asking for trouble. Let’s fix that.

24.5.1 The threat model

For our trading bot, we need to protect against:

  • Unauthorized connections: Someone discovering our EPMD port and brute-forcing(or sniffing) the cookie
  • Eavesdropping: Inter-node traffic(including database queries and trade data) is sent in cleartext by default
  • Port scanning: Exposed EPMD and distribution ports advertising our cluster’s existence

24.5.2 Layer 1: Firewall rules

The first and most important defence is a firewall. Only allow distribution traffic from known node IPs:

# Allow EPMD and distribution ports only from known nodes
sudo ufw allow from 203.0.113.11 to any port 4369
sudo ufw allow from 203.0.113.11 to any port 9100:9155 proto tcp
sudo ufw allow from 203.0.113.12 to any port 4369
sudo ufw allow from 203.0.113.12 to any port 9100:9155 proto tcp
# Deny everything else on these ports (default deny)
sudo ufw enable

This is the single most effective measure. Even if someone obtains your cookie, they can’t connect if the firewall blocks their IP.

24.5.3 Layer 2: A strong cookie

Use the long, random cookie we generated earlier. Avoid anything guessable. The cookie is transmitted during the initial handshake but not used for ongoing encryption - it’s more like a shared password than a session key.

24.5.4 Layer 3: VPN or WireGuard(recommended for production)

For a production deployment, the gold standard is to run all inter-node traffic over a VPN. WireGuard is particularly well-suited for this: it’s fast, simple to configure, and adds encryption with minimal overhead.

With WireGuard, each node gets a private IP(e.g. 10.0.0.1, 10.0.0.2) and all distribution traffic flows through the encrypted tunnel. Your node names use the WireGuard IPs(hedgehog@10.0.0.1), the distribution ports are never exposed to the public internet, and the cookie becomes a secondary safeguard rather than your only protection.

Setting up WireGuard is outside the scope of this chapter, but the configuration is straightforward - typically under 20 lines per node. The WireGuard documentation and the wg-quick tool make it approachable even if you haven’t used a VPN before.

Note on TLS distribution: Erlang also supports encrypting the distribution protocol itself using TLS, via the -proto_dist inet_tls flag. This provides encryption and mutual certificate authentication without a VPN. However, it requires managing TLS certificates on every node, configuring a custom ssl_dist_opt file, and is considerably more complex to set up and maintain than a WireGuard tunnel. If you’re already operating a PKI(Public Key Infrastructure) or are deploying in an environment where VPNs are not an option, TLS distribution is a solid choice. See the Erlang documentation on “Using TLS for Erlang Distribution” for the full setup procedure.

24.5.5 Summary of ports to open

For quick reference, here are the ports each node needs:

Port Protocol Purpose
4369 TCP EPMD(node discovery)
9100-9155 TCP Erlang distribution(inter-node traffic)
4000 TCP Phoenix web server(if using the web UI)

Open these ports only to known node IPs(for 4369 and 9100-9155) or to your management network. Port 4000 can be exposed more broadly if you want external access to the web interface.

24.6 Configuring libcluster

We could connect nodes manually using Node.connect/1, but that’s tedious and fragile. If a node restarts, all its connections are lost and someone (or something) has to reconnect it. libcluster automates this - it continuously attempts to form and heal a cluster based on a configured strategy.

24.6.1 Adding the dependency

# /mix.exs
  defp deps do
    [
      ...
      {:libcluster, "~> 3.5"}, # <= added
      ...
    ]
  end
mix deps.get
...

24.6.2 Choosing a strategy

libcluster ships with several clustering strategies. For VMs with known IP addresses, Cluster.Strategy.Epmd is the right choice - you give it a list of node names and it connects to them on startup. Other strategies exist for dynamic environments(Kubernetes, multicast gossip, DNS), but for a fixed set of VMs, a simple list is the most reliable and debuggable option.

24.6.3 Configuration per environment

In development, we disable clustering or use a local gossip strategy for testing. In production, we use the EPMD strategy with node names from environment variables.

# /config/dev.exs
config :libcluster,
  topologies: []
# /config/test.exs
config :libcluster,
  topologies: []
# /config/integration.exs
config :libcluster,
  topologies: []

For production, we read the list of peer nodes from an environment variable. Add the following inside the if config_env() == :prod do block in runtime.exs:

# /config/runtime.exs (inside the :prod block)
  cluster_nodes =
    case System.get_env("CLUSTER_NODES") do
      nil ->
        []

      nodes_string ->
        nodes_string
        |> String.split(",", trim: true)
        |> Enum.map(&String.trim/1)
        |> Enum.map(&String.to_atom/1)
    end

  config :libcluster,
    topologies: [
      hedgehog: [
        strategy: Cluster.Strategy.Epmd,
        config: [
          hosts: cluster_nodes
        ]
      ]
    ]

The CLUSTER_NODES environment variable is a comma-separated list of node names. On each node, you list all the other nodes it should connect to:

# On node 203.0.113.10
export CLUSTER_NODES="hedgehog@203.0.113.11,hedgehog@203.0.113.12"

# On node 203.0.113.11
export CLUSTER_NODES="hedgehog@203.0.113.10,hedgehog@203.0.113.12"

# On node 203.0.113.12
export CLUSTER_NODES="hedgehog@203.0.113.10,hedgehog@203.0.113.11"

Add CLUSTER_NODES to each node’s /etc/hedgehog/env file.

Note on Erlang’s mesh topology: Erlang clusters are fully connected by default - every node connects to every other node. When node A connects to node B, and node B is already connected to node C, node A will automatically connect to node C as well. This means you technically only need to list one existing node in CLUSTER_NODES - once connected, the new node discovers and connects to all others. However, listing all nodes is more resilient: if the single listed node is down when you start, the new node won’t join the cluster at all.

24.6.4 Adding to the supervision tree

libcluster provides a Cluster.Supervisor that manages the connection strategy. Add it to your supervision tree in application.ex, before the trading supervisors but after the Repo:

# /lib/hedgehog/application.ex
  def start(_type, _args) do
    topologies = Application.get_env(:libcluster, :topologies, [])

    children =
      [
        Hedgehog.Repo,
        {Cluster.Supervisor, [topologies, [name: Hedgehog.ClusterSupervisor]]}, # <= added
        ...
      ] ++ litestream_children()
    ...
  end

The Cluster.Supervisor reads the topology configuration and immediately begins connecting to the listed nodes. If a connection fails, it retries automatically. If a connected node goes down, libcluster detects the disconnection(via Erlang’s built-in node monitoring) and attempts to reconnect.

24.8 Querying data across the cluster

Now for the payoff. We have multiple trading nodes, each with its own SQLite database, connected in a cluster. How do we get a unified view of what’s happening across all of them?

The key insight is that each node already has everything it needs to answer questions about itself - it has a Hedgehog.Repo, schemas, and query functions. All we need is a way to call those functions remotely and merge the results.

24.8.1 Introducing :erpc

Erlang provides the :erpc module(introduced in OTP 23) for executing functions on remote nodes. It improves on the older :rpc module with better error handling and the ability to distinguish between returned values, raised exceptions, and communication failures.

The function we care about is :erpc.multicall/5 - it executes a function on multiple nodes in parallel and collects the results:

iex(hedgehog1@127.0.0.1)1> :erpc.multicall(Node.list(), IO, :puts, ["hello from remote"])
hello from remote
hello from remote
[{:ok, :ok}, {:ok, :ok}]

Each result is tagged with {:ok, value} on success or {:error, reason} on failure. This makes it easy to handle partial results when some nodes are unreachable.

24.8.2 Building a cluster query module

Let’s build a Hedgehog.Cluster module that provides a clean API for cross-node queries:

# /lib/hedgehog/cluster.ex
defmodule Hedgehog.Cluster do
  @moduledoc """
  Query and aggregate data across all connected cluster nodes.
  """

  import Ecto.Query

  alias Hedgehog.Repo
  alias Hedgehog.Strategy.Naive.Settings, as: NaiveSettings

  @default_timeout 5_000

  @doc """
  Returns a list of all connected nodes, including the current one.
  """
  def all_nodes do
    [Node.self() | Node.list()]
  end

  @doc """
  Executes a function on all nodes (including this one) and collects results.
  Failed nodes are excluded from the result.
  """
  def multicall(module, function, args, timeout \\ @default_timeout) do
    nodes = all_nodes()
    results = :erpc.multicall(nodes, module, function, args, timeout)

    nodes
    |> Enum.zip(results)
    |> Enum.flat_map(fn
      {_node, {:ok, value}} when is_list(value) -> value
      {_node, {:ok, value}} -> [value]
      {node, {:error, reason}} ->
        require Logger
        Logger.warning("Cluster call to #{node} failed: #{inspect(reason)}")
        []
    end)
  end

  @doc """
  Executes a function on all nodes and returns results tagged with the node name.
  """
  def multicall_tagged(module, function, args, timeout \\ @default_timeout) do
    nodes = all_nodes()
    results = :erpc.multicall(nodes, module, function, args, timeout)

    nodes
    |> Enum.zip(results)
    |> Enum.map(fn
      {node, {:ok, value}} -> {node, {:ok, value}}
      {node, {:error, reason}} -> {node, {:error, reason}}
    end)
  end
end

24.8.3 Querying orders across the cluster

Now we can build specific query functions. Let’s add a function that returns the trading status of all symbols across every node:

# /lib/hedgehog/cluster.ex (add to the existing module)

  @doc """
  Returns all naive strategy settings with status :on across all nodes.
  """
  def active_trading_symbols do
    multicall(__MODULE__, :local_active_symbols, [])
  end

  @doc false
  def local_active_symbols do
    Repo.all(
      from(s in NaiveSettings,
        where: s.status == :on,
        select: %{symbol: s.symbol, node: ^to_string(Node.self())}
      )
    )
  end

The pattern is simple: define a local_* function that queries the local database, then call it across all nodes via multicall/4. Each node executes the query against its own SQLite database and returns the result. The calling node merges everything into a single list.

Try it from any node in the cluster:

iex(hedgehog1@127.0.0.1)1> Hedgehog.Cluster.active_trading_symbols()
[
  %{symbol: "XRPUSDT", node: "hedgehog1@127.0.0.1"},
  %{symbol: "BTCUSDT", node: "hedgehog2@127.0.0.1"},
  %{symbol: "ETHUSDT", node: "hedgehog2@127.0.0.1"},
  %{symbol: "DOGEUSDT", node: "hedgehog3@127.0.0.1"}
]

One function call, one result, from every node in the cluster. Each node handles its own symbols but you can see the full picture from anywhere.

24.8.4 Adding a cluster status summary

Let’s add one more practical function - a high-level summary of the cluster state that’s useful for monitoring:

# /lib/hedgehog/cluster.ex (add to the existing module)

  @doc """
  Returns a summary of each node's status: name, connected state,
  and number of active trading symbols.
  """
  def status do
    results = multicall_tagged(__MODULE__, :local_status, [])

    Enum.map(results, fn
      {node, {:ok, status}} ->
        Map.put(status, :node, node)

      {node, {:error, reason}} ->
        %{node: node, status: :unreachable, reason: reason}
    end)
  end

  @doc false
  def local_status do
    active_count =
      Repo.aggregate(
        from(s in NaiveSettings, where: s.status == :on),
        :count
      )

    %{
      status: :running,
      version: Application.spec(:hedgehog, :vsn) |> to_string(),
      active_symbols: active_count,
      uptime_seconds: :erlang.statistics(:wall_clock) |> elem(0) |> div(1000)
    }
  end
iex(hedgehog1@127.0.0.1)1> Hedgehog.Cluster.status()
[
  %{node: :"hedgehog1@127.0.0.1", status: :running, version: "0.1.0", active_symbols: 1, uptime_seconds: 3842},
  %{node: :"hedgehog2@127.0.0.1", status: :running, version: "0.1.0", active_symbols: 2, uptime_seconds: 7201},
  %{node: :"hedgehog3@127.0.0.1", status: :running, version: "0.1.0", active_symbols: 1, uptime_seconds: 1923}
]

Notice the version field - we pull it from Application.spec/2, which reads the version compiled into the release. This becomes particularly useful during rolling deploys: if you’ve updated some nodes but not others, status/0 will show you exactly which version each node is running. In the next chapter, it will tell you whether a hot code upgrade has been applied.

Note on :erpc.multicall and timeouts: The timeout in :erpc.multicall/5 is a per-node timeout, not a total timeout. If you set it to 5000ms and have three nodes, the worst case is still ~5 seconds(because calls run in parallel), not 15 seconds. However, if a node is unreachable and TCP hasn’t timed out yet, the call to that node may block for the full timeout duration. Keep this in mind when choosing your timeout value - too short and you’ll get false negatives on slow networks; too long and a single dead node will make your status check feel sluggish.

Note on code consistency: When using :erpc to call functions on remote nodes, the code on the remote node must have the module and function you’re calling. This is obvious when all nodes run the same release, but it becomes important during rolling deployments - if node A has been updated with a new version of Hedgehog.Cluster that adds a function, but node B is still running the old version, the :erpc call to node B will fail with an UndefinedFunctionError. Keep this in mind when deploying updates across a live cluster. We’ll look at strategies for managing this in the next chapter when we discuss hot code reloading.

24.10 Troubleshooting

Clustering problems are almost always one of four things. Here’s how to diagnose each:

Nodes can’t connect at all: Check that EPMD is running on both machines(epmd -names), that port 4369 is reachable from the other node(nc -zv 203.0.113.11 4369), and that the distribution port range(9100-9155) is open. Also verify that both nodes use the same naming mode(-name, not -sname).

Cookie mismatch: If the cookie doesn’t match, you’ll see a log message like connection attempt from disallowed node. Verify cookies match with Node.get_cookie() on each node.

DNS resolution failures: If you use DNS names instead of IP addresses in your node names, every node must be able to resolve every other node’s hostname. Test with nslookup or dig from each machine.

Nodes connect then immediately disconnect: This usually means a network timeout or a firewall rule that allows the initial TCP handshake but drops the connection shortly after. Check for stateful firewall rules and ensure the distribution port(not just EPMD) is whitelisted.

24.11 Final thoughts

We now have a distributed trading system. Multiple independent nodes, each running its own trading strategies against its own local SQLite database, connected via Erlang’s native distribution protocol. Any node can query any other node’s data with a single function call. Nodes can join and leave the cluster dynamically - libcluster handles reconnection automatically.

The architecture is deliberately simple. There’s no central coordinator, no message broker, no shared database. Each node is self-sufficient: it can trade on its own even if every other node in the cluster goes down. The cluster adds visibility and coordination, not dependency.

Notice what we didn’t do: we didn’t build consensus protocols, distributed transactions, or shared state. Each node owns its own data. The Hedgehog.Cluster module is a read-only aggregation layer - it queries remote nodes but never writes to them. This keeps the system simple and avoids the hardest problems in distributed computing. If you need a node to change its behavior, you connect to it directly and tell it what to do.

The security setup is layered: firewall rules block unauthorized access, a strong random cookie prevents accidental connections, and a VPN(WireGuard) encrypts everything in transit. You don’t need all three layers, but each one reduces your attack surface.

We also made several choices in this chapter that might have seemed like minor details but were actually laying groundwork. The strip_beams: false option preserves the metadata OTP needs to compute code differences. The tarball-based deployment keeps old version directories intact alongside new ones. The version field in Cluster.status/0 tells you exactly what each node is running. These pieces all come together in the next chapter.

What’s next?

We have a running cluster, but deploying updates currently means stopping each node, unpacking the new tarball, and starting it again. For a trading bot, even a few seconds of downtime means missed trades. In the next chapter, we’ll use OTP’s hot code reloading to upgrade a running node without stopping it - push new code, generate an appup describing how to transition each module’s state, and install it on the live system while trades continue to execute. The release structure we built in this chapter is what makes that possible.
