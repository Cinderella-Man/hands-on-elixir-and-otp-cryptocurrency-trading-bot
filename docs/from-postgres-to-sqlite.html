Hands-on Elixir & OTP: Cryptocurrency Trading Bot

Chapter 23 From Postgres to SQLite

In the last chapter, we consolidated six umbrella apps into a single Phoenix application - one repo, one database, one config surface. The structure is clean. But if you look at how it runs, there’s still an external process standing between our application and its data: Postgres, managed via Docker.

For development, that means running docker compose up -d before you can do anything. For deployment, it means each trading node needs a network path to a shared Postgres instance - and each write is a round-trip over that link. If the database goes down or the network hiccups, all nodes stop trading.

For a system we intend to distribute across independent nodes, this is exactly the wrong architecture. We want each node to be self-contained: its own processes, its own state, its own persistence. SQLite gives us that. It’s an embedded database that lives as a single file alongside the application - no server process, no network dependency, no Docker container. If that sounds like a toy database, it isn’t - SQLite handles more read traffic than most Postgres deployments will ever see, and for a single-node writer like our trading bot, it’s more than enough. The ecto_sqlite3 adapter makes the switch almost transparent to our application code; Ecto’s Repo API stays the same, migrations need only minor adjustments.

In this chapter, we’ll swap Postgres for SQLite, remove the Docker dependency entirely, and replace the ecto_enum library(which relies on Postgres-specific CREATE TYPE statements) with Ecto’s built-in Ecto.Enum. We’ll also set up Litestream for continuous database backups to S3-compatible storage - because an embedded database without off-site replication is one disk failure away from total data loss. By the end, our trading bot will be a fully self-contained application that carries its database with it - which is exactly what we’ll need when we introduce clustering in the next chapter.

23.1 Objectives

  • removing the Docker/Postgres dependency
  • swapping the database adapter to SQLite
  • replacing ecto_enum with Ecto.Enum
  • updating status values from strings to atoms
  • updating migrations for SQLite compatibility
  • setting up continuous backups with Litestream and S3-compatible storage

23.2 Removing the Docker dependency

The docker-compose.yml file at the root of our project exists solely to run Postgres. Since we’re moving to SQLite, we no longer need it:

rm docker-compose.yml

We should also add SQLite’s database files to .gitignore. SQLite uses up to three files per database (the main .db file, a write-ahead log -wal, and a shared-memory file -shm):

# /.gitignore
# Database files
*.db
*.db-shm
*.db-wal

23.3 Swapping the database adapter

23.3.1 Dependencies

Inside mix.exs, we need to swap postgrex for ecto_sqlite3 and remove ecto_enum (we’ll replace it with Ecto’s built-in Ecto.Enum in the next section):

# /mix.exs
  defp deps do
    [
      ...
      {:ecto_sqlite3, "~> 0.22"}, # <= added (replaces postgrex)
      ...
    ]
  end

Remove the following lines from deps:

# /mix.exs
  defp deps do
    [
      ...
      {:postgrex, ">= 0.0.0"},  # <= remove
      ...
      {:ecto_enum, "~> 1.4"},   # <= remove
      ...
    ]
  end

23.3.2 Repo adapter

Next, we need to update Hedgehog.Repo to use the SQLite3 adapter:

# /lib/hedgehog/repo.ex
defmodule Hedgehog.Repo do
  use Ecto.Repo,
    otp_app: :hedgehog,
    adapter: Ecto.Adapters.SQLite3 # <= updated
end

23.3.3 Configuration

The Postgres configuration used hostname, username, password, and a database name. SQLite only needs a file path. We need to update all environment config files.

Starting with the development configuration:

# /config/dev.exs
config :hedgehog, Hedgehog.Repo,
  database: Path.expand("../hedgehog_dev.db", Path.dirname(__ENV__.file)),
  pool_size: 5,
  show_sensitive_data_on_connection_error: true

SQLite supports multiple concurrent readers in WAL mode, so a pool size of 5 is a reasonable default - enough to allow several read queries to overlap. Writes are serialized regardless of pool size; SQLite permits only one writer at a time. A larger pool won’t speed up writes, but it can help if you have many concurrent reads.

Note: Path.expand("../hedgehog_dev.db", Path.dirname(__ENV__.file)) resolves to a file in the project root directory. This is the standard convention for Phoenix projects using SQLite.

The test configuration follows the same pattern but uses a separate database file:

# /config/test.exs
config :hedgehog, Hedgehog.Repo,
  database: Path.expand("../hedgehog_test.db", Path.dirname(__ENV__.file)),
  pool_size: 5,
  pool: Ecto.Adapters.SQL.Sandbox

And the integration configuration:

# /config/integration.exs
config :hedgehog, Hedgehog.Repo,
  database: Path.expand("../hedgehog_integration.db", Path.dirname(__ENV__.file)),
  pool_size: 5,
  pool: Ecto.Adapters.SQL.Sandbox

Finally, the runtime configuration for production. We replace the DATABASE_URL environment variable with DATABASE_PATH:

# /config/runtime.exs
if config_env() == :prod do
  database_path =
    System.get_env("DATABASE_PATH") ||
      raise """
      environment variable DATABASE_PATH is missing.
      For example: /data/hedgehog/hedgehog.db
      """
  # remove: maybe_ipv6 = ....

  config :hedgehog, Hedgehog.Repo,
    database: database_path,
    pool_size: String.to_integer(System.get_env("POOL_SIZE") || "5")
  ...

Note regarding releases: In dev.exs, we used Path.expand relative to the current file. In runtime.exs (Production), we rely on the DATABASE_PATH environment variable. This is critical because when you build an OTP Release, the configuration files are evaluated, but the source code directories may not exist on the target server. Always use absolute paths or environment variables for database locations in production.

Note on concurrency: The ecto_sqlite3 adapter defaults to WAL (Write-Ahead Logging) mode, which allows concurrent reads while a write is in progress. However, SQLite still permits only one writer at a time. If two processes attempt to write simultaneously, one will wait up to busy_timeout milliseconds (default: 2000ms) before raising an error.

For a trading bot where multiple GenServers may write to the database, you might want to increase this value in your Repo configuration. The default 2000ms is fine for our current setup, but if you add more concurrent writers later, bumping it prevents spurious timeouts:

# /config/*.exs
    config :hedgehog, Hedgehog.Repo,
      ...
      busy_timeout: 5000

See the ecto_sqlite3 adapter documentation for the full list of tuning options.

At this point, our adapter is configured but our migrations and schemas still reference ecto_enum types that only work with Postgres. Let’s fix that next.

23.4 Replacing ecto_enum with Ecto.Enum

This is the most involved part of the migration - not because it’s difficult, but because the ecto_enum library is woven into three separate areas of our codebase: schema files, migration files, and dedicated enum modules.

23.4.1 Why the change is necessary

The ecto_enum library uses Postgres-specific CREATE TYPE statements to define custom enum types at the database level. SQLite has no equivalent feature - it stores everything as one of five types: NULL, INTEGER, REAL, TEXT, or BLOB.

Ecto’s built-in Ecto.Enum takes a different approach: it stores enum values as plain strings in the database and handles the atom conversion in the application layer. This works identically across all database backends.

23.4.2 Removing the enum modules

We have three *_status_enum.ex files that use defenum from the ecto_enum library. Since Ecto.Enum is declared inline within schema definitions, these dedicated modules are no longer needed. We can delete all three:

rm lib/hedgehog/streamer/settings_status_enum.ex
rm lib/hedgehog/strategy/naive/settings_status_enum.ex
rm lib/hedgehog/data/collector/settings_status_enum.ex

23.4.3 Updating the schema files

Each settings schema currently references its companion enum module. We need to replace those references with Ecto.Enum field declarations.

Starting with the streamer settings:

# /lib/hedgehog/streamer/settings.ex
defmodule Hedgehog.Streamer.Settings do
  use Ecto.Schema

  # remove: alias Hedgehog.Streamer.SettingsStatusEnum

  ...

  schema "streamer_settings" do
    ...
    field(:status, Ecto.Enum, values: [:on, :off]) # <= updated
    ...
  end
end

Next, the naive strategy settings - note that this enum has three values including :shutdown:

# /lib/hedgehog/strategy/naive/settings.ex
defmodule Hedgehog.Strategy.Naive.Settings do
  use Ecto.Schema

  # remove: alias Hedgehog.Strategy.Naive.SettingsStatusEnum

  ...

  schema "naive_strategy_settings" do
    ...
    field(:status, Ecto.Enum, values: [:on, :off, :shutdown]) # <= updated
    ...
  end
end

And finally, the collector settings:

# /lib/hedgehog/data/collector/settings.ex
defmodule Hedgehog.Data.Collector.Settings do
  use Ecto.Schema

  # remove: alias Hedgehog.Data.Collector.SettingsStatusEnum

  ...

  schema "collector_settings" do
    ...
    field(:status, Ecto.Enum, values: [:on, :off]) # <= updated
    ...
  end
end

23.4.4 Important: strings must become atoms

There is one catch. The old ecto_enum library was lenient - it happily accepted both strings ("on") and atoms (:on). Ecto.Enum is stricter: it only accepts atoms. Any place in our codebase that passes a string like "on" or "off" to an Ecto query or an insert_all call will now raise an Ecto.ChangeError at runtime.

We need to find and update every occurrence. There are three areas to fix: autostart queries, status update functions, and the seed script.

23.4.5 Autostart queries

Three supervisor modules query for settings where: s.status == "on". We need to change the string "on" to the atom :on in each:

# /lib/hedgehog/streamer/binance/dynamic_streamer_supervisor.ex
  def autostart_workers do
    Repo.all(
      from(s in Settings,
        where: s.status == :on, # <= updated from "on" to :on
        select: s.symbol
      )
    )
    ...
# /lib/hedgehog/strategy/naive/dynamic_trader_supervisor.ex
  def autostart_workers do
    Repo.all(
      from(s in Settings,
        where: s.status == :on, # <= updated from "on" to :on

    ...
# /lib/hedgehog/data/collector/dynamic_worker_supervisor.ex
  def autostart_workers do
    Repo.all(
      from(s in Settings,
        where: s.status == :on, # <= updated from "on" to :on
        select: s.topic

    ...

23.4.6 Status update functions

Several modules have update_status/2 functions that pass string status values when building changesets or structs. We need to update these to atoms as well:

23.7 Backing up the database

With Postgres, backups were someone else’s problem - your hosting provider, a cron job running pg_dump, or Docker volume snapshots. With SQLite, the database is a file sitting in our project root. That simplicity is the whole point - but it also means backups are entirely our responsibility.

You might think cp hedgehog_dev.db hedgehog_backup.db would do the trick, but it won’t. If the application is running and a write is in progress, a raw file copy can produce a corrupted backup. SQLite’s WAL mode makes this even trickier: the database state is split across the main .db file and the -wal file, and copying them at different moments can produce an inconsistent snapshot. Even if you could copy them atomically, a local backup on the same disk isn’t really a backup - if the disk fails, you lose both.

SQLite does provide safe one-shot backup mechanisms(VACUUM INTO or the .backup command), but those produce point-in-time snapshots. For a trading bot where every executed trade matters, losing hours of data to a periodic snapshot is not acceptable. What we need is continuous replication to remote storage.

Litestream does exactly this. It monitors the SQLite WAL file and streams every change to S3-compatible storage in near-real-time - replication lag is measured in seconds.

The litestream Hex package wraps the Litestream binary in a supervised GenServer. It downloads the binary automatically on first start, manages it as an Erlang port, and integrates cleanly with our supervision tree. Let’s set it up.

23.7.2 Configuring Litestream

Litestream needs to know which Repo to back up, where to replicate it, and how to authenticate. We’ll configure these per environment.

For development and test, we disable Litestream entirely - there’s no reason to replicate a throwaway dev database to remote storage:

# /config/dev.exs
config :hedgehog, Hedgehog.Litestream, enabled: false
# /config/test.exs
config :hedgehog, Hedgehog.Litestream, enabled: false
# /config/integration.exs
config :hedgehog, Hedgehog.Litestream, enabled: false

For production, the credentials come from environment variables. Add the following inside the existing if config_env() == :prod do block in runtime.exs:

# /config/runtime.exs (inside the :prod block)
  replica_url =
    System.get_env("LITESTREAM_REPLICA_URL") ||
      raise """
      environment variable LITESTREAM_REPLICA_URL is missing.
      For example: s3://hedgehog/db?endpoint=<account-id>.r2.cloudflarestorage.com
      """

  config :hedgehog, Hedgehog.Litestream,
    enabled: true,
    repo: Hedgehog.Repo,
    replica_url: replica_url,
    access_key_id: System.fetch_env!("LITESTREAM_ACCESS_KEY_ID"),
    secret_access_key: System.fetch_env!("LITESTREAM_SECRET_ACCESS_KEY")

Note: We use Cloudflare R2 in this chapter, but Litestream works with any S3-compatible storage. Backblaze B2, DigitalOcean Spaces, MinIO, and AWS S3 all work - just adjust the LITESTREAM_REPLICA_URL endpoint accordingly.

23.7.3 Adding to the supervision tree

We want Litestream to start automatically in production and stay out of the way in development. Update application.ex to conditionally include it:

# /lib/hedgehog/application.ex
  def start(_type, _args) do
    children =
      [
        ...
      ] ++ litestream_children()
    ...
  end

  defp litestream_children do
    config = Application.get_env(:hedgehog, Hedgehog.Litestream, [])

    if Keyword.get(config, :enabled, false) do
      opts = Keyword.drop(config, [:enabled])
      [{Litestream, opts}]
    else
      []
    end
  end

The litestream_children/0 function returns an empty list when backups are disabled, so the Litestream GenServer simply isn’t started in dev or test. In production, it starts right after the Repo and begins replicating immediately.

When the GenServer starts for the first time, it downloads the Litestream binary appropriate for your platform. Subsequent starts use the cached binary. Once running, Litestream watches the WAL file and continuously pushes changes to your S3 bucket. You don’t need to think about it again - it just runs in the background as a supervised process, and the BEAM restarts it if it crashes.

23.7.4 Setting up the storage bucket

Before deploying, you’ll need an S3-compatible storage bucket. We’ll use Cloudflare R2 because the setup is minimal and the free tier is generous(10 GB storage, zero egress fees). Any S3-compatible service works - Backblaze B2, DigitalOcean Spaces, MinIO - but R2 requires the least ceremony to get started.

Litestream speaks the S3 protocol, so it needs three things from your storage provider: an endpoint(where to connect), an Access Key ID and a Secret Access Key(how to authenticate). For Cloudflare R2, you’ll need to:

  1. Create a storage bucket - this is the container for your backup data. Pick a name that identifies the project(e.g. hedgehog-backups).
  2. Create an API token with read and write permissions scoped to that bucket. This generates an Access Key ID and Secret Access Key pair - save both immediately, as the secret is only displayed once.
  3. Find your Account ID - a unique identifier for your Cloudflare account, visible on the R2 overview page. This becomes part of the endpoint URL.

The replica URL combines the bucket name, a path prefix inside the bucket, and the endpoint. The format is:

https://<account-id>.r2.cloudflarestorage.com/<bucket-name>/<path>

Note that the endpoint uses the bare hostname - no https:// prefix. The <path> (we use db) is just a key prefix inside the bucket; you can name it whatever makes sense for your project.

Set all three environment variables before starting your application:

export LITESTREAM_REPLICA_URL="s3://hedgehog/db?endpoint=<your-account-id>.r2.cloudflarestorage.com"
export LITESTREAM_ACCESS_KEY_ID=<your-r2-access-key-id>
export LITESTREAM_SECRET_ACCESS_KEY=<your-r2-secret-access-key>

23.7.5 Restoring from a backup

If your node’s database is lost (disk failure, corrupted file, fresh deployment), you can restore the latest replica using the Litestream CLI.

Install the Litestream binary that matches the version bundled by the Hex package(run mix deps and look at the litestream entry to confirm the binary version):

curl -fsSL https://github.com/benbjohnson/litestream/releases/download/v0.3.13/litestream-v0.3.13-linux-amd64.tar.gz \
  | tar -xz -C /usr/local/bin

Then restore:

export LITESTREAM_ACCESS_KEY_ID=<your-r2-access-key-id>
export LITESTREAM_SECRET_ACCESS_KEY=<your-r2-secret-access-key>
litestream restore -o /data/hedgehog/hedgehog.db \
  "s3://hedgehog-backups/db?endpoint=<your-account-id>.r2.cloudflarestorage.com"

Start your application normally after the restore. Litestream will pick up where it left off and resume replication.

Note: For automated deployments, you’ll typically add a restore step to your startup script that runs before the application boots. This ensures a fresh node always recovers the latest database state from S3 before accepting traffic.

23.8 Final thoughts

This was a short chapter, but the operational impact is significant. We replaced Postgres with SQLite, eliminating the Docker dependency entirely. We swapped ecto_enum(Postgres-specific) for Ecto.Enum(database-agnostic) and updated status values from strings to atoms across queries, update functions, and seeds. We rewrote all migrations to use plain string columns instead of custom Postgres types - and preserved all existing tests and trading functionality without touching the business logic.

Notice what didn’t change: the Repo API usage, the supervision tree, the trading logic. What did change were the status value literals - "on" to :on, "off" to :off - because Ecto.Enum is stricter than the old ecto_enum library about types. That strictness is a feature: it catches type mismatches at query time instead of silently inserting bad data.

The trade-off is real, though. SQLite doesn’t support concurrent writes from multiple processes as gracefully as Postgres does. For our use case - a trading bot running as a single node writing its own local data - that’s perfectly fine. We’re not building a multi-tenant web app; we’re building independent trading nodes.

And that’s exactly where we’re headed.

What’s next?

We now have an application with no external infrastructure dependencies - no database server to manage, no Docker container to run, just our Elixir app and a local database file. This is the perfect building block for distribution. In the next chapter, we’ll configure libcluster for automatic cluster formation and start running multiple trading nodes side by side. Each node will carry its own SQLite database, stream its own symbols, and execute its own trades - while remaining connected via Erlang’s distribution protocol for coordination and monitoring.
